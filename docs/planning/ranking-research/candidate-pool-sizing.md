# Candidate-Pool Sizing for the Clustering Stage — Evidence Memo

Research memo for the check-recipe ranking engine ([hypothesis P6](../../architecture/ranking-engine.md#presentation--budget)). Topic: how large the candidate set feeding the stage-5 clustering step should be, given that today it is the top `per_page = 20` slice of the similarity ranking — an inherited pagination default doing double duty as the clustering pool ([stage 3](../../architecture/ranking-engine.md#stage-3--the-candidate-window)). Constraint inherited from principle 2: displayed similarity scores stay raw and nothing is hidden from flat pagination — a pool rule shapes only the clustered summary's composition, so relative/score-based sizing is admissible here where a display cutoff is not. Researched 2026-07-17.

## 1. What production two-stage systems feed their second stage

The dominant pattern across retrieve-then-refine architectures is a fixed absolute pool in the low hundreds, with ~100 as the strong mode, feeding a second stage that returns single digits. The Sentence-Transformers retrieve & re-rank reference pipeline — the RAG-era canonical description — sizes it explicitly: "we first use a retrieval system that retrieves a large list of e.g. 100 possible hits which are potentially relevant for the query," and justifies the number by second-stage cost: "Scoring thousands or millions of (query, document)-pairs would be rather slow. Hence, we use the retriever to create a set of e.g. 100 possible candidates which are then re-ranked by the Cross-Encoder." ([sbert.net, Retrieve & Re-Rank](https://sbert.net/examples/applications/retrieve_rerank/README.html))

Diversity-specific second stages use the same order of magnitude or smaller. LangChain's MMR — the most widely deployed diversify-over-retrieval default — fetches a pool five times the display size: `k: int = 4, fetch_k: int = 20`, where `fetch_k` is the "Number of `Document` objects to fetch to pass to MMR algorithm." ([langchain_core/vectorstores/base.py](https://raw.githubusercontent.com/langchain-ai/langchain/master/libs/core/langchain_core/vectorstores/base.py)) It is worth noticing that our current 20 equals MMR's *pool* default — but MMR clusters/selects 4 from it, while we ask 20 to stand in for the whole corpus. Elasticsearch's diversified sampler — "a filtering aggregation used to limit any sub aggregations' processing to a sample of the top-scoring documents" — defaults to the same hundred: "The `shard_size` parameter limits how many top-scoring documents are collected in the sample processed on each shard. The default value is 100." ([Elasticsearch diversified sampler aggregation](https://www.elastic.co/guide/en/elasticsearch/reference/current/search-aggregations-bucket-diversified-sampler-aggregation.html)) Vespa structures the same idea as a phase boundary: the diversity element exists "to filter out non-diverse results before second-phase ranking (and during match-phase if used)," with "min-groups … the minimum number of different values of that attribute that should be included in the result set" ([Vespa result diversity](https://docs.vespa.ai/en/querying/result-diversity.html)) — diversity is enforced on the candidate set *before* the expensive stage, exactly the P6 shape.

So field practice for retrieve-N → display-k ratios runs from 5:1 (MMR) through 20–30:1 (cross-encoder top-100 → top 3–5). Our current ratio is ~7:1 (20 → 3 exemplars) — inside the MMR band but built from a pagination constant, not a diversity decision; a 100 → 3 pool would put us at ~33:1, the top of the conventional range.

## 2. Fixed absolute vs relative sizing — and the operator's two instincts

There is a real literature on *not* fixing the cutoff. Ranked-list truncation is a named task — "the problem of determining how many results to return" has "received less attention despite being of critical importance," balancing "the overall relevance, or usefulness of the results, with the user cost of processing more results" ([Choppy, arXiv:2004.13012](https://arxiv.org/abs/2004.13012)). The recent RAG-flavored version is Adaptive-k: "Sort the scores in descending order. Compute their first discrete differences and choose the index k where the similarity drop is the largest," a method that "does not require model fine-tuning, extra LLM inferences or changes to existing retriever–reader pipelines." ([Adaptive-k, arXiv:2506.08479](https://arxiv.org/html/2506.08479)) Its successor makes the case *against* any global constant: "retrieval scores exhibit strong query-dependent variability, particularly in large and heterogeneous corpora," so "a single global cutoff is inherently brittle, frequently including irrelevant contexts for some queries while excluding critical information for others" — and their knee-detection + tail-stability method reaches "near-oracle retrieval quality (F1 within 2–3% of oracle)." ([Tail-Aware Adaptive-k, arXiv:2606.11907](https://arxiv.org/html/2606.11907v1))

Against that evidence, the operator's two instincts (task brief, 2026-07-17): *"Is it ok to just have the same 10000 cap that the clustering stage has? That feels like it might then just have too many irrelevant things in it. Maybe we just grab the top 30% or something?"*

- **The 10000 cap: the suspicion is right.** A whole-corpus pool is what the map and briefing use *because their query is "everything"* — there is no relevance gradient to protect. On a query surface, Tail-Aware's finding applies directly: an unbounded pool guarantees "including irrelevant contexts" for every specific query, and k-means has no relevance term — centroids are pulled by mass, so at 3 clusters over thousands of mostly-irrelevant candidates the exemplars describe the corpus, not the query. It also has a concrete cost cliff in our stack (§4).
- **The top-30%: right instinct, wrong denominator.** Wanting *relative* sizing is what the adaptive-k literature endorses — but percent-of-matches is not the relative quantity it endorses. In our engine every readable recipe "matches" (stage 1 orders **all** candidates; there is no threshold), so top-X%-of-matches degenerates to X%-of-corpus: ~300–900 today, ~12,000 at benchmark scale — growing exactly when the operator's own diagnosis says the top gets *more* homogeneous, and unbounded in cost. The literature's relative signal is the **score distribution** (largest gap, knee, tail stability), which is corpus-size-independent and per-query — and admissible for us because it only shapes the clustered summary.

## 3. What clustering-over-retrieval specifically wants

Search-result clustering engines have published pool guidance directly. Carrot2 (Lingo — the reference implementation of the scatter/gather idea in modern form): "A hundred documents on input is probably the minimum for any statistical significance of features discovered by the algorithm," "an optimum number of documents would probably fall between 100 and a few thousands," and "Carrot2 should successfully deal with up to a thousand of documents, a few paragraphs each" ([Carrot2 docs, Tuning clustering](https://carrot2.github.io/release/4.3.0/doc/tuning-clustering/)); past that, "For more than about 1000 documents, Lingo clustering will take a long time and large memory." ([Carrot2 docs, Choosing the algorithm](https://carrot2.github.io/release/4.3.1/doc/choosing-clustering-algorithm/)) Note our current 20 is 5× below Carrot2's *minimum* for meaningful cluster structure. The both-sided failure mode is the same one the operator named: too small and the pool is homogeneous (clusters split hairs — no diversity to find); too large and cluster centroids answer "what is this corpus about" instead of "what bears on this query." The 100–1000 band is where search-result clustering has historically lived.

## 4. Cost realities in this stack

The retrieval plan, not k-means, is the binding constraint. The ANN path streams top-k from HNSW with `k = Math.max(ANN_CANDIDATE_MIN, (offset + limit) * ANN_DEDUPE_MARGIN)` where `ANN_CANDIDATE_MIN = 60`, `ANN_DEDUPE_MARGIN = 3`, and `ANN_CANDIDATE_MAX = 400` — "Above ~400 the planner flips back to the exhaustive plan anyway (measured), so past this we just run the exact scan directly." Measured on the real corpus: "8-18ms at k=40-100 vs 50-65ms exhaustive warm — and cold it touches ~k vectors' pages instead of the whole table, which removes the after-idle latency cliff." ([vector-search.service.ts](../../../apps/backend/src/services/vector-search.service.ts)) Solving for the pool: **pool ≤ 133 stays ANN** (133 × 3 = 399); pool ≥ 134 forces the exhaustive scan on every check, whose cost grows with corpus size and reintroduces the cold cliff. A 10000 or percent-of-corpus pool pays that on every check, permanently.

Clustering cost scales fine if we borrow the map's trick: the check path currently clusters at full 3,072 dims (it never sets `vectorDims`), while the map/briefing surfaces MRL-truncate to `MAP_VECTOR_DIMS = 768` — "4× less transfer/parse/k-means work than the full 3,072 dims with near-identical 2D-layout quality. Search is unaffected (it runs in SQL on the full stored vectors)." ([search-pipeline.ts](../../../apps/backend/src/services/search-pipeline.ts)) A pool-size lever should adopt the truncation in the same change. Prior art inside the repo already brackets the answer: the eval runner uses `perPage: 100, // whole corpus — no-cutoff retrieval, golden sets are small` ([ranking-eval.ts](../../../apps/backend/src/eval/ranking-eval.ts)), and the map's `perPage: 10000, // no pagination limit — clustering IS the pagination` ([traces.ts](../../../apps/backend/src/routes/traces.ts)) is the no-query-surface case, not a precedent for the check path.

## Recommendations for this system

**Default proposal: a fixed absolute `clusterPoolSize = 100`, as a `RankingConfig` lever decoupled from `per_page`, shipped default-off (off ⇒ pool = `per_page`, byte-stable legacy).** Rationale: 100 is simultaneously (a) the two-stage-retrieval convention (sbert, Elasticsearch diversified sampler default), (b) Carrot2's stated statistical minimum for meaningful cluster structure, (c) already the eval runner's number, and (d) comfortably inside the ANN sweet spot (k = 300 ≤ 400 — checks stay on the 8-18ms plan). It changes only what k-means sees; flat/`expand` mode, pagination arithmetic, displayed scores, and the result *set* are untouched. Ship `vectorDims = 768` for check-path clustering in the same lever (measured map precedent), so the 5× pool costs ~1.25× the current vector work.

**Alternatives worth sweeping in the harness:**
1. **Size sweep: 20 (control) / 60 / 100 / 133 / 400.** 133 is the exact ANN boundary; 400 measures whether anything past the boundary is worth an exhaustive scan. Expect diminishing aspect-coverage returns past ~100 on 1-3k corpora.
2. **Bounded score-gap adaptive pool: largest-gap index clamped to [20, 133]** (Adaptive-k's first-discrete-difference rule, floor = today's behavior, cap = ANN boundary). The literature's answer to query-dependent variability, admissible here because it shapes only the summary; costs one sort we already have.
3. **Score-relative pool: all candidates within δ of the top score, same [20, 133] clamp**, δ swept (e.g. 0.05/0.10/0.15 cosine). Simpler to explain to agents than gap detection; degenerates gracefully on flat score distributions via the clamp.

**What settles it:** aspect coverage (Candidate C in the [serendipity memo](serendipity-diversity-metrics.md)) should rise with pool size — that is P6's core prediction; genuine-recall on the golden sets must not fall (exemplars drifting off-query would show up here first — this is the 10000-cap failure mode made measurable); tau/nDCG guardrails must hold within the noise thresholds; and per-check latency at the swept sizes confirms the ANN-boundary math. If adaptive variants beat fixed-100 on coverage *without* a genuine-recall cost on the small-corpus datasets, prefer the bounded gap rule; otherwise fixed-100 wins on explainability. Either way the flip ships through the standard sweep → report → ruling → versioned-changelog path.
