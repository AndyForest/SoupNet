# ClaimNet Engineering Principles

Core principles guiding every design and implementation decision. Treat these as constraints, not preferences. When in doubt, raise the tension explicitly rather than silently violating one.

---

## 1. Test-Driven Design (TDD)

Write the test first. Then write the code that passes it.

**Why it matters here:** ClaimNet is a system whose value depends on correctness at boundaries — the contract between frontend and backend, the ranking formula, the access control rules, the embedding pipeline. Bugs at these seams are expensive. Tests make the contract explicit before the implementation exists.

**In practice:**
- New service functions: write the test first, then implement
- New endpoints: write integration tests first (happy path + auth failure + validation error)
- Ranking logic: property-based tests that verify score ordering invariants
- Schema changes: migration tests that verify the schema can be migrated forward and backward

**Tools in use:** Vitest for unit + integration. Integration tests hit the running Docker backend on :3101 — no Supertest, no testcontainers. `npm run test:ci` spins a fresh isolated Postgres on :5534 to reproduce CI exactly. See `docs/testing-plan.md`.

---

## 2. Tracer Bullets (Vertical Slices First)

Before building wide, build thin and deep. Implement complete paths from the user/agent to the database and back before adding breadth.

> "Fire a tracer bullet through the system to see where it lands, then adjust aim." — *The Pragmatic Programmer*

**For ClaimNet, the first tracer bullet is:**

1. **Search-as-logging:** User creates a recipe book → gets a search URL with embedded API key → AI agent visits the search page → submits a trace (claim + evidence + references) via GET → trace is logged → vector embedding computed → search results returned. This cuts through the entire system: web page → endpoint → service → Drizzle write → vector pipeline → hybrid search → response.

**What this finds early:**
- Three-entity schema correctness (traces, evidence, references, linking tables)
- Vector pipeline integration with new source types
- GET-based form encoding limits and workarounds
- API key scoping and recipe-book access
- Search result ranking with coverage signals

Build this flow end-to-end (with stub implementations where needed) before building any other features.

---

## 3. Composable, Testable Modules with Rich Functionality

Every module should do one thing well, expose a typed interface at its boundary, and be internally rich enough to serve its purpose without leaking concerns. No business logic in route handlers or React components.

**The vector pipeline is the canonical example.** The 4-table embedding pipeline (`embedding_sources` → `embedding_chunk_strategies` → `embedding_chunks` → `embedding_vectors`) is a fully abstracted, self-contained module. It supports multiple chunking strategies, multiple embedding models, multiple task types, N:N relationships between content and vectors, and tsvector-based lexical search — all behind a clean interface. The domain model (traces, evidence, references) never knows about chunking strategies or vector dimensions. This abstraction is a core design principle: modules should be internally sophisticated where that sophistication serves the product, while presenting a simple interface to consumers.

**Layer rules:**

```
Route handler (Hono)  →  calls service  (no business logic here)
Service               →  calls repository + calls other services
Repository            →  calls Drizzle directly
```

**Frontend:**
```
Page / route component  →  calls React Query hook (generated by Orval)
Hook                    →  calls API (generated from OpenAPI spec)
UI component            →  receives props, emits events, no API calls
```

Storybook is the registry of composable frontend components with interactive examples. It serves both AI coding assistants (structured, predictable interfaces to assemble) and human developers. Each component should be demonstratable in isolation without a running backend.

**Testing at boundaries:**
- Services and repositories are tested with real Drizzle queries against the running Docker backend
- HTTP routes are tested by sending real requests to :3101 (or the isolated :5534 DB via `npm run test:ci`)
- MCP tools are tested end-to-end via the `soupnet-local` MCP server against the same backend (`docs/testing-plan.md` Layer 4b)
- Frontend component tests are still sparse — Storybook-driven interaction tests remain aspirational

## 4. Deterministic Code Generation — Don't Write What Can Be Generated

Prefer generating code from a single source of truth over maintaining parallel representations manually.

| Source of truth | What's generated |
|---|---|
| Zod schemas in `packages/contracts` | OpenAPI 3.1 spec (`/api/openapi.json`) |
| OpenAPI spec | React Query hooks + TypeScript types (Orval → `packages/api-client`) |
| Drizzle schema (`packages/db/src/schema/`) | SQL migrations (`drizzle-kit generate`) |
| Drizzle schema | TypeScript types (Drizzle inference) |
| MCP tool `inputSchema` (Zod) | MCP tool JSON Schema (MCP SDK generates from Zod) |

**Implications:**
- Never hand-write API client code — Orval generates it
- Never hand-write the OpenAPI spec — Zod registry generates it
- Never write TypeScript types that duplicate Drizzle table shapes — infer them
- Never write SQL DDL by hand — `drizzle-kit generate` from schema

**Schema SSOT:** `packages/db/src/schema/` is the single source of truth for all database tables. One schema (`claimnet`), one migration system (Drizzle).

## 5. Work in Priority Sequence — Solve Unknown Unknowns First

Build in the order that surfaces integration risks earliest. Don't build auth before you can run an end-to-end search. Don't build the admin panel before the core data model is stable.

**Phase ordering:**
1. Schema + migrations → know what the DB looks like
2. Tracer bullet flows → know where the seams break
3. Auth + ACL → know what's protected
4. Core features → build on a known-good foundation
5. Optimizations → only after you know what needs optimizing

## 6. ACID Where It Matters, Async Where It's Safe

**Synchronous (within a transaction):**
- User signup → personal org creation (org must exist before user is returned)
- Recipe check → trace + evidence + references insert + sync full_document embedding + hybrid search (atomic)
- Any operation where the caller needs to know it succeeded before returning

**Asynchronous (via in-process pg-boss consumers, per ADR-0020):**
- Experimental-strategy chunking + embedding (6 parallel strategies, batched up to 200/call against Gemini limits)
- Email delivery (verification, invitation, password reset)
- Moderation scanning (planned)

Embeddings NEVER block primary writes on the async path. The sync path writes a `full_document` vector at check time so the first response has semantic search; experimental strategies fill in asynchronously and appear on subsequent checks. Multimodal embeddings are sync-only (ADR-0019) because the async `VectorCheckItem` shape doesn't carry file bytes — enforced at the `enqueueEmbedding` boundary.

## 7. Explicit Access Control — Never Implicit

Access control has **two strictly separate credential populations**, matching the two client populations:

- **Human SPA users → JWT only.** Issued at login, stored in `localStorage`, verified by `requireAuth` middleware. 7-day lifetime, stateless, one symmetric secret.
- **AI agents → API key only.** Issued by a human via the Keys page, hashed at rest, verified by API-key middleware. Scoped to recipe books, short-lived (daily) or bounded (scoped), revocable individually.

**The two populations never mix.** An AI agent never holds a JWT — agents authenticate exclusively with API keys. This is a deliberate, load-bearing security boundary: API keys are recipe-book-scoped, short-lived, append-only by design, and revocable without touching the owning human's session. A compromised agent should cost the user one key, not their account. Routes that accept JWT do not fall back to API keys and vice versa; cross-presenting either credential to the wrong surface returns 401.

Rules — **JWT-protected (human SPA) routes:**

- Every route that reads user data must use `requireAuth` middleware **paired with `requireVerifiedEmail`** (the secure-by-default convention). Apply both globally on the router and opt *out* specific routes — never opt in. The only opt-outs today are `GET /auth/me` and `POST /auth/resend-verification`, because they must be reachable pre-verification to render the verify-pending state; each new opt-out is a new pre-verification attack surface and must be justified in a code comment.
- System-level routes add `requireSystem` middleware (role = 'system') on top of the above.
- Service functions accept `userId` from the verified JWT — never trust client-supplied IDs.
- JWT payload carries `sub` + `role` as the authorization inputs. Do not base authorization on `c.user.email` — the email is a staleness hazard (a 7-day-old token still carries the user's old address if they changed it).

Rules — **API-key-protected (agent) routes:**

- API key validation checks expiry, revocation, and recipe-book scope on every request. (At the schema level, scope is stored as `read_group_ids` / `write_group_ids` arrays per the deferred Group → Recipe Book schema rename — see ADR-0016.)
- Write operations additionally check the key's write-recipe-book set; read operations check the read-recipe-book set. Keys never grant access to recipe books they were not scoped to at creation.
- JWT routes and API-key routes are mounted on disjoint paths — the same handler never accepts both. Helpers that sit between the two (e.g., services that need "current user" context) take the resolved principal as a typed argument, not a raw header.

Auth is designed to be swappable: `users.provider` + `users.external_id` support future OIDC federation on the JWT side. API-key issuance is independent of the human auth provider.

Failure-mode reasoning: when a new protected route is added and the author forgets to wire middleware, the default convention makes it fail *closed* (401/403) rather than open. Opt-in auth is the pattern that led to F15-class findings in the 2026-04-09 audit.

## 8. Never Make Direct Database Edits — And Never Hand-Write Migrations

Always use Drizzle migrations to change the schema. Never run ALTER TABLE, INSERT, or other DDL/DML directly against the database. Never write a migration SQL file from scratch — always start with `drizzle-kit generate`.

**Standard workflow:**
- Edit schema files in `packages/db/src/schema/`
- Run `npx drizzle-kit generate` in `packages/db/` to create a migration
- Migrations are applied automatically at backend startup
- To reset locally: `docker compose down -v` and restart fresh

**For data backfills** (e.g. populating new columns from existing data, seed data for system roles):
- Run `drizzle-kit generate` first to get the structural ALTER TABLE statements + matching snapshot
- **Edit the generated `.sql` file** to add UPDATE/INSERT statements between the structural changes
- The snapshot tracks structural state only — adding data manipulation to a generated migration is safe and idiomatic
- Use `ON CONFLICT DO NOTHING` for idempotency on seeds
- For larger backfills, split into two migrations: one for ADD + backfill, one for DROP after the backfill is verified

**For initial seed data** (default settings, system roles):
- `drizzle-kit generate --custom` produces an empty migration
- Add INSERT statements with `ON CONFLICT DO NOTHING`
- Drizzle tracks it like any other migration — runs once, on fresh installs and existing DBs

**Anti-patterns:**
- ❌ Hand-writing a migration `.sql` file from scratch — breaks the snapshot chain because the snapshot won't be regenerated, causing future `drizzle-kit generate` calls to fail with collision errors
- ❌ Putting backfill data in `/scripts/` — scripts have no version tracking, no automatic application, no rollback. Fresh installs will be broken
- ❌ Propagating broken snapshots by copying them forward to silence collision errors — investigate the root cause (almost always a missing UUID update in `id`/`prevId`)

**The snapshot vs SQL vs runtime distinction:**

| Artifact | Used by | When |
|---|---|---|
| `migrations/*.sql` | `migrate()` runtime | Backend startup |
| `migrations/meta/_journal.json` | Both | Both |
| `migrations/meta/*_snapshot.json` | `drizzle-kit generate` only | Build time only |
| `__drizzle_migrations` table | `migrate()` runtime | Backend startup tracking |

A broken snapshot doesn't prevent migrations from running — it only prevents `drizzle-kit generate` from working for new migrations. This means snapshot bugs can lurk undetected until the next person tries to add a column.

**CI guardrail:** `npx drizzle-kit check` should run in CI. It catches snapshot collisions in <1 second and prevents this class of bug from merging. (Backlog: add to `.github/workflows/ci.yml`.)

**Drizzle versions:** drizzle-orm@0.45+ requires drizzle-kit@0.31+. Older drizzle-orm versions break drizzle-kit's introspect command.

## 9. Defer What You Can Measure, Don't Optimise What You Haven't Profiled

The ranking weights in `packages/domain/src/ranking.ts` are hypotheses. The embedding model choices are initial bets. The chunk sizes in the vector store are guesses. None of these should be optimised until there's data.

What "data" means here:
- Real retrieval quality scores from users/agents
- Embedding latency and cost measurements
- Search relevance feedback from validations
- DB query profiling under realistic load

## 10. SVG Is Dangerous — Sanitize Before Serving

SVG can contain embedded JavaScript (`<script>`, `onload=`, `href=javascript:`). Any SVG that passes through the system must be sanitized before being served to any client (browser or agent). Even with Content-Security-Policy headers on the backend, these files are downloaded and potentially rendered by remote agent clients where CSP doesn't apply.

Plan: define a strict subset of the SVG type system (safe elements and attributes), either enforce at upload or strip to that subset. Not in scope for v1 — `image/svg+xml` is on the allowlist but should be removed until the sanitization pipeline is audited.

## 11. Configuration Lives In One Place Per Environment (Aspirational)

Each environment should have exactly one source of truth for configuration. Code should provide sensible defaults where possible, and surface required values loudly when they're missing — not silently fall back to localhost or fabricate URLs.

**Target state, per environment:**

| Environment | Single source of truth | Read by |
|---|---|---|
| Local dev | `.env` | Backend, worker, tests, local Docker Compose |
| CI | `env:` block in `.github/workflows/ci.yml` | Quality-gate test run only |
| Hosted dev / staging / prod | Per-environment config (non-secret env vars or infra-as-code variables) + a secrets store (secret) | The deployed runtime, frontend build, deploy job |

**Drift to watch for in any deployment platform:**

- Per-environment values inside infrastructure-as-code modules should be variables, not hardcoded — so the module is reusable across environments without forking.
- Deploy-job workflows should source environment-specific names from infrastructure outputs (or platform variables wired from those outputs), so adding a new environment is a config change rather than a workflow fork.
- The PG\* / `DATABASE_URL` connection helpers in `apps/backend/src/db.ts` have zero defaults — every required var throws if missing. That's fine as a posture (fail loudly > mystery localhost), but it means `.env.example` files are the de facto schema documentation and have to be hand-synced.

**Why this matters:** every additional config tier is another place that drifts. A future `prod` environment that requires editing an infrastructure module, a per-environment variables file, the CI workflow, and a manual secrets-store step is one where someone forgets a tier and ships a broken deploy. Consolidating tiers is cheap when you have one environment and expensive once you have several.

**Not a blocker for current work** — note as drift, fix in a focused refactor, don't bundle into unrelated changes.

## 12. Admin Pages Are Data-Model-Driven, Not Curated Reports

Admin pages should mirror the underlying data model — surface aggregates over real tables, drill into individual records, and let the system admin investigate from the same vantage point as a `psql` session would. Resist the urge to build curated "executive summary" reports that hide where the data lives.

**Why:** Curated reports answer the questions you anticipated. Real investigation requires the questions you didn't anticipate. If the admin page only shows the metrics the developer thought were important, the system admin will end up `psql`-ing the database to debug anything else. That's a sign the admin page failed.

**In practice:**
- **Top-level: schema overview.** Show row counts per table. The admin should be able to see at a glance what data exists.
- **Per-table: aggregates + full row access.** Show the distributions and group-bys that matter, AND a paginated/filterable view of the raw rows. Both, not either-or.
- **Per-row: drill-in showing every column.** No hiding fields. JSON columns get pretty-printed. The admin sees what the database stores.
- **Filters that map to columns.** Let the admin filter by the same dimensions the database has — queue name, state, retry count, age — not by curated categories.
- **Read-only first.** Build the visibility before the actions. Once you know what the admin actually needs to do, add targeted operations (retry, requeue, delete) — not bulk dashboards full of buttons.

**What this rules out:**
- "KPI dashboards" with cherry-picked numbers and no way to see the underlying data
- Hiding columns because they're "not interesting" — the admin decides what's interesting
- Building actions before visibility — actions without observability create risk
- Mixing application logic into the admin view (e.g., business rules that hide certain rows) — show what's there, let the admin interpret

**Examples in the codebase:**
- `/admin/queues` — mirrors the `pgboss.*` schema. Tables list, queue config table, jobs explorer with full filter/sort/drill-in. Looks like a database explorer, not a dashboard.
- `/admin/workers/embeddings` — application-specific layer ON TOP of the data view. Shows strategy coverage (the domain interpretation), but the underlying vector status counts are visible too. Action buttons (retry) come AFTER the visibility is in place.

**Two layers of admin pages:**
- **Generic / data-model layer** (`/admin/queues`) — independent of what the application does. Anyone using pg-boss could use this page. Could be extracted as a library.
- **Application-specific layer** (`/admin/workers/*`) — domain interpretation of the same underlying data. For embeddings: strategy coverage, retry semantics, error grouping. The two layers complement each other — neither replaces the other.

## 13. AI Agents Are First-Class Clients

AI agents are THE primary client of ClaimNet. The web search page is designed for agent consumption first, human consumption second.

**What this means in practice:**
- The **web search page** (GET form, minimal HTML, embedded instructions) is the primary agent interface. Any AI agent with web browsing can use ClaimNet with zero configuration.
- The **MCP server** is the secondary interface for platforms where MCP provides lower friction. Both hit the same backend.
- Agent-facing pages minimize token cost: semantic HTML, no inline styles, linked CSS for human rendering. Instructions reference concepts agents already know (stigmergy, Design Thinking) and link to authoritative sources for precision.
- Search page instructions are written for a sophisticated, state-of-the-art AI reader. Concise, with hyperlinks for depth — like a research paper, not a tutorial.
- We test, measure, and compare friction across the AI agent ecosystem (web page vs. MCP vs. REST) and optimize for whatever agents actually prefer.

**What this rules out:**
- Designing from the human UI perspective and treating agent access as an afterthought
- Requiring complex authentication flows for basic search (API key in URL is sufficient)
- Token-heavy pages with decorative markup that agents must parse

## 14. All Outgoing Email Is Logged — Metadata Only, 60-Day Retention

Every email the system sends goes through the logged sender (`sendLoggedMail` in `apps/backend/src/services/email.service.ts`), which writes one row per send — success or failure — to `claimnet.email_log`.

**Why:** The email log is three things at once: the light CRM (who did we contact, when, about what — surfaced on `/admin/emails` with filter/sort/pagination), the deliverability record (failed sends are visible instead of vanishing into a rejected promise), and the security/abuse review surface (a compromised account or runaway code path that sends email leaves a trail an auditor can sweep in one query).

**In practice:**
- **New email kinds go through `sendLoggedMail`.** Never call `transporter.sendMail` directly. Add the kind to the `EmailKind` union so the log stays queryable by type.
- **Metadata only — never bodies.** Bodies carry secrets (verification links, reset tokens, invite tokens). A leaked email log must not be a token archive. Log recipient, kind, subject, status, error.
- **60-day retention** (`EMAIL_LOG_RETENTION_DAYS`), purged opportunistically by the sender itself before each send — no scheduler to maintain, and the table stays bounded as long as email flows at all. Disclosed in the privacy policy §8.
- **Send failures still log** (status `failed` + error text), and the failure still propagates to the caller — logging never swallows errors, and a logging failure never blocks the send.

**What this rules out:**
- Fire-and-forget sends with no record (the pre-2026-06-11 state)
- Storing rendered email bodies "for debugging"
- Unbounded retention — the log is operational, not an archive
