# Transcript-mining report — session "soup-net-voice-positioning" (audited 2026-06-10, session work 2026-05-27/28)

> Self-audit per the transcript-mining briefing. Session scope: landing-page rewrite (hero copy, two pillars, walkthrough section with three steps + +1 coda panel, CTA, sticky bar), design-thinking.md updates (Archetype 5 AI-reluctant collaborator, Two-layer UVP split, "team" → "group/collaborator" sweep), three new solution illustrations + briefing format, sidebar nav add, responsive mobile fix. ~4 recipe checks across the session; many judgment calls unchecked. Self-audit bias acknowledged; a fresh session can re-run this against the transcript on disk for a stronger pass.

## Finding: rapid-iteration mode systematically suppressed checks
- **Type:** briefing-friction (the session's biggest miss)
- **Situation (scrubbed):** A long session of word-level copy iteration with fast operator feedback — heading went through five versions in succession ("How Soup.net recipe checks solve this" → "Both promises run on the recipe check" → "Both promises, one mechanism" → "Both run on the recipe check" → "How a recipe check works"), each rewrite responding to a precise operator critique. Zero of the five versions were recipe-checked. The same pattern held for the pillar prose rewrites, the +1 coda framing, and most of the CTA candidate set. Late in the session the operator interjected *"recipe check as we go remember"* — an explicit signal that the rhythm had drifted.
- **What a check would have needed to return:** at minimum the prior session's exemplar showing landing-page heading choices have load-bearing corpus history (vendor-independence framing, autonomy hook, agency-centeredness). Some of these would have raised counter-framings I would have surfaced earlier rather than discovering mid-iteration.
- **Suggested improvement:** the briefing's "When to check" lists three moments (starting a task, judgment call, completed work) that assume well-spaced decisions. It does not address *iteration mode* — many small judgment calls in quick succession during conversational refinement. Candidate KB/briefing addition: an "iteration mode" rhythm — when refining the same artifact across many rounds, check the framing call (not every word change), and check before committing the final form. Adding the explicit cue "if you've made three rewrites without checking, you're in iteration mode — check the framing" would have caught this session's drift early.

(This matches the prior session's "components checked, synthesis unchecked" finding — same failure mode, different surface.)

## Finding: image-caption check ran post-hoc instead of pre-apply
- **Type:** timing
- **Situation (scrubbed):** The agent recommended adding italic captions under the pillar illustrations to bridge problem-imagery to answer-text, drafted three caption candidates per image, and applied the operator's pick — then recipe-checked the underlying caption-bridging *strategy* as part of the cross-session "recipe check as we go" reminder, after the captions were already shipped. The check returned useful corpus guidance ("gentle and knowing humor rather than overtly punchline-based" from a 2026-04-05 designer recipe) that would have informed the candidate captions, plus a forward-looking note about a richer earlier Pillar 2 concept ("fragments of one larger shared map") that the current artwork didn't fully realize. Both were useful but late — the captions were already in the file.
- **What the check returned:** voice corroboration plus a forward-reference. Same retrieval would have improved candidate generation if performed before drafting.
- **Suggested improvement:** briefing/KB candidate — when generating candidate copy/visual options, the candidate-generation moment is itself a judgment call (which voice, which framings to include in the set). Checking before generation reshapes the option space; checking after only validates the chosen survivor. Candidate cue for the briefing: "if you're about to draft three+ options, check the framing first — it shapes which options enter the candidate set."

## Finding: option-presentation pattern wasn't mapped to divergent-checks
- **Type:** briefing-friction
- **Situation (scrubbed):** Throughout the session the agent repeatedly presented 3-5 candidate options to the operator (hero subhead options A/B/C; CTA options P/Q/R/S; pillar format A/B/C; heading variants). The briefing's "Divergent recipe checks" section addresses this pattern for web-only agents (clickable URLs) and MCP agents (elicitation, sub-agent, sequential confirmation). The agent intuitively followed the MCP-agent guidance ("present the options as text in your reply and wait for the user to pick before calling check_recipe on the chosen one") but did NOT always check the chosen survivor. The briefing's pattern explicitly says to check only after the user picks; the agent often skipped that step.
- **What a check would have needed to return:** for the heading choice, prior corpus framing on landing-page section titles (the prior session's "How a recipe check works" framing already existed in the corpus at 71% similar). Each unchecked survivor passed up an opportunity to ratify or refine.
- **Suggested improvement:** briefing edit candidate — the "Divergent recipe checks" section currently emphasizes the option-generation discipline (don't check candidates the user hasn't picked). Add a complementary instruction: "after the user picks, check the survivor — that's where the recorded judgment lives." Adding "the chosen survivor is the recipe to record" as a closing sentence would tighten the loop.

## Finding: candidate-list checks suppress the "presenting them is also a judgment" beat
- **Type:** unchecked-judgment (granularity, related to above)
- **Situation (scrubbed):** The agent often generated 3-4 options *of its own design* (heading variants, caption candidates, CTA candidates) and surfaced them. The act of choosing which options to include in the set is itself a judgment call — what framings did the agent decide were plausible and which did it silently exclude? None of these option-set-curation moments were recipe-checked. The session presented dozens of options; the union of options-the-agent-considered-but-didn't-show is much larger than what the operator saw.
- **What a check would have needed to return:** prior corpus framing on which option families this product's positioning typically explores (e.g., autonomy-anchored, action-anchored, agency-anchored — these are real prior categories the corpus would have surfaced for CTA candidates).
- **Suggested improvement:** companion to the previous finding — checking before option generation reshapes the candidate set; checking only the survivor validates the choice within a set the agent unilaterally bounded. The richer pattern: "Check the framing at session start (selects the option family), then check the survivor at choice time (records the call)." This is two checks per branching moment, not one.

## Net pattern across findings

Three of the eight unchecked-or-mistimed moments share a root cause: **rapid conversational iteration suppresses the checking instinct.** The briefing's three "when to check" moments assume distinct decision points; iteration mode is one long continuous decision with many small commits, and the briefing has no rhythm for it. The operator's explicit "recipe check as we go" reminder confirmed the systemic miss — that reminder is the smoke; the briefing's quiet on iteration mode is the fire.

Secondary pattern: when the agent generates options and presents them to the operator, *both* the option-set generation *and* the operator's choice are judgment calls. The briefing's divergent-checks section addresses the choice timing but not the set-generation. Both ends of the option-presentation pattern leak unchecked calls.

Successes were concentrated at session-start framing decisions (agent-first repositioning, AI-reluctant archetype, CTA candidate). When the check happened, it returned material value — the corpus is alive and well; the failure mode is purely "didn't check."
